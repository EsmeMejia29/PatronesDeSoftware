Patrones de Diseño de Software

Patrón Iterator

El patrón Iterator permite acceder secuencialmente a los elementos de un objeto agregado sin exponer su representación subyacente.

El Problema

Imagina que estás construyendo una plataforma de e-commerce y tienes un listado de productos almacenados internamente en un árbol binario complejo para optimizar las búsquedas rápidas. En otra parte del sistema, tienes un listado de promociones guardado en una simple matriz asociativa (array).

¿Qué duele aquí? Si el cliente (por ejemplo, el generador de facturas o la vista del carrito) necesita recorrer ambos listados para calcular un total, estás obligado a exponer la estructura interna de tus colecciones. El código cliente se llena de bucles for, foreach recursivos para el árbol, y lógicas pesadas que cambian drásticamente según cómo guardes los datos. Si mañana decides cambiar el árbol binario por una lista enlazada, romperás todo el código cliente que dependía de esa estructura.

La Solución

El patrón extrae el comportamiento de recorrido de la colección y lo delega a un objeto independiente llamado Iterator. Este objeto encapsula todos los detalles del recorrido (como la posición actual y cuántos elementos quedan), exponiendo una interfaz unificada y ultra simple. El cliente ya no necesita saber si recorre un árbol, una base de datos o un array; solo le pide al iterador el "siguiente" elemento.

Estructura de Componentes

  • Iterator: Define las operaciones necesarias para recorrer la colección (obtener el elemento actual, avanzar al siguiente, verificar si hay más).
  • ConcreteIterator: Implementa el recorrido específico para una estructura de datos concreta y mantiene el estado de la iteración actual.
  • Aggregate (Iterable): Define una interfaz para crear y retornar su iterador correspondiente.
  • ConcreteAggregate: Mantiene la colección de datos real y expone el método para fabricar el iterador configurado para su estructura.

Ejemplo de Código (PHP)

Estructura mínima que demuestra cómo desacoplar el recorrido de una colección personalizada:

<?php
// 1. La interfaz del Iterador
interface MyIterator {
    public function current();
    public function next(): void;
    public function hasNext(): bool;
}

// 2. El Iterador Concreto
class ArrayIterator implements MyIterator {
    private int $position = 0;
    public function __construct(private array $items) {}

    public function current() { return $this->items[$this->position]; }
    public function next(): void { $this->position++; }
    public function hasNext(): bool { return isset($this->items[$this->position]); }
}

// 3. La Colección que expone el Iterador
class ProductCollection {
    private array $products = [];
    public function addProduct(string $product): void { $this->products[] = $product; }
    public function getIterator(): MyIterator { return new ArrayIterator($this->products); }
}

// 4. Uso por el cliente (completamente desacoplado de la estructura interna)
$collection = new ProductCollection();
$collection->addProduct("Laptop");
$collection->addProduct("Mouse");

$iterator = $collection->getIterator();
while ($iterator->hasNext()) {
    echo $iterator->current() . "\n";
    $iterator->next();
}

Señales de cuando SÍ aplica:

  • Colecciones complejas bajo el capó: Cuando tu estructura de datos es compleja (como árboles, grafos o respuestas paginadas de una API) y quieres ocultar esa complejidad al cliente.
  • Múltiples formas de recorrido: Cuando necesitas recorrer la misma estructura de diferentes maneras (por ejemplo, recorrer un árbol de forma pre-orden, in-orden o post-orden) sin duplicar lógica en el cliente.
  • Polimorfismo en recorridos: Cuando quieres que tu código sea capaz de iterar sobre diferentes tipos de colecciones de manera uniforme sin importar cuál sea.

Señales de cuando NO aplica (Sobre-ingeniería):

  • Colecciones lineales simples: Si tu aplicación solo maneja listas simples o arrays planos de PHP y el lenguaje ya provee un bucle nativo estructurado (como un simple foreach).
  • Acceso puramente aleatorio: Si tu lógica requiere saltar constantemente de forma aleatoria a posiciones específicas (como acceder directamente al índice 15, luego al 2) en lugar de un recorrido secuencial.
  • Restricciones severas de rendimiento: En sistemas embebidos o procesos críticos donde instanciar un objeto iterador intermedio añada una sobrecarga de memoria innecesaria.

Ejemplo Real en la Industria

Este patrón es tan fundamental que la mayoría de lenguajes modernos lo traen integrado en su núcleo. En el ecosistema de PHP, contamos con la interfaz nativa Iterator y IteratorAggregate.

Un ejemplo real y verificable se encuentra en las Lazy Collections de Laravel (implementadas mediante Illuminate\Support\LazyCollection). Cuando procesas millones de registros de una base de datos utilizando el ORM Eloquent, cargarlos todos en memoria saturaría el servidor.

Laravel soluciona esto implementando IteratorAggregate apoyado por generadores de PHP (usando yield). Esto permite recorrer gigabytes de registros de la base de datos secuencialmente usando un simple bucle foreach, manteniendo en memoria únicamente un solo registro a la vez: 

use App\Models\User;

// Laravel crea un iterador bajo el capó mediante cursores de base de datos
$users = User::cursor(); 

// El cliente itera millones de filas de forma segura, sin colapsar la RAM
foreach ($users as $user) {
    echo $user->email;
}
